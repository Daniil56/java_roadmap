## Для кого этот модуль

**Минимум:** модуль 5 завершён --- есть опыт работы со Spring Boot, базой данных (PostgreSQL) и основами Docker. Этого достаточно, чтобы упаковать проект в CV и начать откликаться на вакансии.

**Идеально:** знакомство с микросервисной архитектурой, Kafka и Kubernetes (модули 6--8). Чем шире стек --- тем больше инженерных кейсов для CV и рассказа на собеседовании.

Также модуль полезен тем, кто пришёл в roadmap с опытом, но не может упаковать свои знания в убедительный карьерный сигнал: есть стек, есть проект, есть решения, но HR не отвечает на отклики, CV не продаёт опыт, а рассказ о проекте на собеседовании сводится к перечислению технологий.

---

## Зачем этот модуль

До этого места roadmap строил систему: Java Core, Gradle, тесты, Docker, PostgreSQL, Spring Boot, REST API. Как минимум --- полноценное приложение с persistence и контейнеризацией. Как максимум --- микросервисный стек с Kafka, Kubernetes и observability. Задача меняется: не "дописать ещё один сервис", а уметь предъявить построенное --- через CV, через рассказ о проекте, через понимание того, как работает командная разработка.

**Как устроен поиск работы на ру-рынке:** кандидат публикует CV через HeadHunter. HR либо не отвечает, либо зовёт на собеседование. Собеседование либо быстро заканчивается, либо даёт отрицательный результат. Ссылка на GitHub-репозиторий в CV никому не нужна --- рекрутеры и техлиды смотрят на то, как кандидат описывает опыт, а не на код.

**Проблема, которую решает модуль:** CV содержит перечень технологий вместо инженерных кейсов, рассказ о проекте сводится к "использовал Spring Boot и Kafka", а на вопросы про командную работу и Agile --- тишина.

**Что происходит на технических секциях в 2026:** формат сдвинулся от теоритческих вопросов и задач на чистые алгоритмы к приближённым к реальной работе задачам. Минимум --- code review: найти баги в чужом коде, объяснить проблемы, предложить рефакторинг. Максимум --- live coding в своей IDE: от простых задач (leetcode-уровень) до проектирования системы по описанию и реализации на Spring. Также набирает силу тренд на ИИ-навыки: как кандидат использует AI-агентов, как верифицирует AI-код, как описывает правила (CLAUDE.md, .rules .skils mcp).

**Результат модуля:**

- CV упаковывает опыт в инженерные кейсы с измеримыми результатами
- проект можно уверенно рассказать за 60 секунд и за 7 минут
- есть понимание Agile, ролей в команде и Code Review --- то, что спрашивают на каждом собеседовании
- есть подготовка к реальным форматам техсекций: live coding, code review чужого кода, system design
- есть понимание, как использовать AI-инструменты в работе и верифицировать AI-код
- проведены mock interview с конкретными доработками

## FR и NFR

> Общая рамка: [README -- Как работать с требованиями](../README.MD#как-работать-с-требованиями).

### Функциональные требования

**CV и упаковка опыта:**

- CV содержит главный проект как инженерный кейс с измеримыми результатами
- Для каждого проекта описан контекст, стек, вклад и результат
- Достижения сформулированы по формуле: действие + технология/паттерн + измеримый результат
- CV проходит ATS-скрининг: ключевые слова из целевых вакансий присутствуют в тексте

**Рассказ о проекте:**

- Подготовлена короткая версия (60--90 секунд): что за система, стек, эволюция, 1--2 ключевых решения
- Подготовлена длинная версия (5--7 минут): контекст, архитектура, эволюция, инженерные кейсы, качество, что улучшил бы
- Обе версии отрепетированы и не противоречат друг другу

**Инженерные кейсы:**

- Подготовлены 2--3 инженерных кейса для интервью
- Каждый кейс содержит: контекст, проблему, альтернативы, решение, компромиссы, способ проверки
- Для каждого кейса есть артефакт-подтверждение (код, конфигурация, диаграмма, тест)

**Командная работа и Agile:**

- Понимание Agile-методологий: Scrum (спринты, planning, daily, retrospective, review) и Kanban (WIP-лимиты, flow)
- Знание ролей в команде: разработчик, тимлид, QA, аналитик, DevOps --- и как они взаимодействуют
- Понимание Code Review-процесса: цель, типичные замечания, как отвечать на замечания
- Знание branching-стратегий: trunk-based, GitFlow, feature branches --- и когда какая уместна
- Умение оценивать и декомпозировать задачи
- Коммуникация в команде: постановка вопросов, синхронизация, эскалация блокеров

**Mock interview:**

- Проведены 2--3 mock interview или самостоятельных прогона
- Зафиксированы замечания, слабые места и список доработок

**Технические секции собеседований:**

- Понимание форматов live coding: задача с обсуждением (30--45 мин), короткие задачи "пулемётом", код с багами --- найди ошибку
- Умение проходить code review секцию: чтение чужого кода, обнаружение багов, рефакторинг по SOLID, коммуникация замечаний
- Аккуратность на edge cases: пустой массив, null, переполнение, пограничные условия --- важнее алгоритмической гениальности
- Знание Java-контрактов: equals/hashCode, сериализация, generics, concurrency-примитивы
- Понимание system design на уровне Middle: API дизайн, модели данных, выбор БД, оценка нагрузки
- Умение верифицировать собственное решение: написать тесты на edge cases, проверить корректность

**ИИ в работе разработчика:**

- Опыт работы с AI-инструментами: бесплатный CLI-агент (Qwen Code, OpenCode или Kilo) или чат (DeepSeek, Qwen Chat) --- доступны в РФ без VPN
- Понимание MCP-экосистемы: подключение MCP-плагинов (PostgreSQL, Kubernetes, Kafka, Grafana) к AI-агенту для доступа к контексту проекта — навык, накопленный в модулях 4--8
- Умение ставить задачу AI-агенту: специфичный промпт с контекстом, доменом, ограничениями
- Умение описывать правила для AI: проектные инструкции (AGENTS.md, CLAUDE.md, .rules) --- стек, команды, антипаттерны
- Навык верификации AI-кода: критический review сгенерированного кода, edge cases, безопасность
- Понимание рисков AI-генерированного кода: SQL injection, утечки данных, правдоподобный но неверный код
- Подготовка к AI-assisted интервью: умение использовать AI-инструмент в real-time для анализа кода, генерации тестов, рефакторинга

### Нефункциональные требования

- CV не содержит невербализуемых утверждений: каждое достижение можно раскрыть на собеседовании конкретным примером
- Инженерные кейсы звучат как решения, а не как buzzwords: есть контекст, альтернатива, выбранный путь, компромисс
- Mock interview выявляют конкретные пробелы, а не абстрактное "надо больше готовиться"

### Как сформулировать достижение

Каждое достижение строится по формуле: **действие + технология/паттерн + измеримый результат или контекст**.

| Слабая формулировка | Сильная формулировка |
|---|---|
| Использовал Spring Boot | Спроектировал REST API на Spring Boot с 15+ эндпоинтами, RBAC и OpenAPI-документацией |
| Писал тесты | Поддерживал покрытие тестами >= 80% (JaCoCo) с unit, integration и API-тестами |
| Работал с Docker | Контейнеризировал приложение и БД в docker compose с healthchecks и volume для данных |
| Делал микросервисы | Разделил монолит на 4 микросервиса по DDD bounded contexts с отдельными БД и WireMock-тестами |
| Использовал Kafka | Реализовал Saga-паттерн через Kafka для распределённой операции регистрации с идемпотентным consumer |

### Инженерные кейсы для интервью

Инженерный кейс --- мост между кодом и собеседованием. Хороший кейс показывает не "что делал", а "почему сделал именно так".

**Структура кейса:**

- **Контекст:** где в проекте возникла задача
- **Проблема:** что не устраивало в исходном состоянии
- **Альтернативы:** какие варианты рассматривались
- **Решение:** что выбрано и почему
- **Компромиссы:** что потеряли, чем заплатили
- **Проверка:** как убедились, что решение работает

**Типы кейсов, которые хорошо работают на собеседованиях:**

- переход от монолита к микросервисам по DDD bounded contexts
- появление persistence layer и миграций (JDBC -> JPA)
- вынос синхронных вызовов в асинхронный контур через Kafka
- подключение observability-стека (метрики, трейсинг, логи)
- стабилизация ненадёжного сценария через Retry + Circuit Breaker

## Перед стартом

- Как минимум модуль 5 завершён: есть Spring Boot-приложение с БД и Docker
- Желательно знакомство с микросервисами и Kafka (модули 6--8), но не обязательно
- Проект можно запустить целиком или воспроизвести ключевую часть сценария
- Есть 2--3 технических решения, которые реально принимались в ходе работы над проектом
- Определён тип целевых вакансий (junior/middle backend, стек, домен)

Если хотя бы один пункт не выполнен --- карьерная упаковка будет опираться на неполные артефакты.

## Как проверить результат

| Артефакт | Что должно быть на выходе | Как проверить |
|---|---|---|
| CV | Проект описан как инженерный кейс с измеримыми результатами | Есть контекст, стек, вклад, результат |
| Короткий рассказ | 60--90 секунд, покрывает суть проекта | Проговаривается без чтения с листа |
| Длинный рассказ | 5--7 минут, есть логичный порядок и переходы | Покрывает архитектуру, эволюцию, кейсы, качество |
| Инженерные кейсы | 2--3 кейса по шаблону | На каждый кейс есть артефакт-подтверждение |
| Блок Agile и командная работа | Понимание процессов, ролей, Code Review | Устный рассказ без заглядывания в шпаргалку |
| Live coding подготовка | Решены 5--10 типовых задач с фокусом на edge cases | Задачи решены корректно, edge cases обработаны |
| Code review практика | Проведён review чужого кода с фиксацией замечаний | Найдены баги, предложен рефакторинг |
| AI-инструменты | Опыт работы с хотя бы одним AI-инструментом (CLI-агент или чат) | Сгенерированный код верифицирован, риски понятны |
| Mock interview | 2--3 прогона с заметками | Зафиксированы правки и доработки |
| README проекта | Понятен без устных пояснений | Новый читатель может ответить: что это, зачем, как запустить |

**Модуль считается пройденным, если:**

- CV упаковывает главный проект как инженерный кейс с измеримыми результатами
- рассказ о проекте не сводится к перечислению технологий
- есть 2--3 инженерных кейса, привязанных к артефактам
- понимание Agile, ролей в команде и Code Review позволяет уверенно отвечать на собеседовании
- есть подготовка к live coding, code review секциям и system design
- есть опыт работы с AI-инструментами и верификации AI-кода
- после mock interview внесены конкретные правки
- README проекта можно прочитать и понять без устных пояснений
