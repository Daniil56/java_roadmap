## Зачем этот модуль

Цель — не «выучить синтаксис», а заложить **инженерную базу**: даже консольный проект собирается, тестируется и проходит style checks.

Тесты и style checks нужны здесь потому, что на маленьком проекте это дёшево и понятно. Более тяжёлые quality-практики и расширенный AI-tooling осознанно выносятся в другие модули, чтобы не перегружать старт.

**Результат модуля — консольный сервис записи на мероприятие**, который станет основой для эволюции через все следующие модули.

Сервис решает конкретную проблему: организатору нужно управлять регистрациями, не отслеживая их вручную. FR/NFR описывают поведение системы с точки зрения действий стейкхолдеров (организатор, участник). Первый вопрос при проектировании: «какую проблему мы решаем?» — а не «какие классы создать?».

AI на этом этапе — проводник по концепциям и ошибкам, а не генератор кода. Задаём вопросы, разбираем, почему код ведёт себя так, а не иначе, читаем ошибки компиляции — но код пишем сами. Этот подход (Ask-Don't-Copy) определяет, как AI будет использоваться во всех следующих модулях.

Общие правила работы с AI и FR/NFR — в [корневой README roadmap](../README.md).

## FR и NFR

> Общая рамка: [README → Как работать с требованиями](../README.md#как-работать-с-требованиями). Здесь — применение к консольному сервису этого модуля.

### Функциональные требования (что делает система)

- Создавать мероприятие с ограничением по участникам
- Регистрировать участника, если есть свободные места
- Отменять регистрацию и освобождать место
- Выводить список участников мероприятия
- Отображать консольное меню с нумерованными действиями
- Собирать ввод пользователя и делегировать выполнение в сервисный слой

### Алгоритмические требования

| Операция | Сложность |
|----------|-----------|
| Поиск участника по ID | O(n) — допустима |
| Добавление участника | O(1) |
| Удаление участника | O(n) — допустима |
| Вывод списка | O(n) |

Данные хранятся в памяти, без внешних файлов. Память растёт линейно с количеством участников.

### Нефункциональные требования (как проверяем качество)

- Код проходит Checkstyle
- Есть базовые unit-тесты на критичную логику
- Сборка воспроизводима (`./gradlew build` работает локально)
- AI помогает разбирать концепции и понимать ошибки, а не генерирует код (Ask-Don't-Copy)
- Каждая строка кода объяснима — на вопрос «зачем эта строка?» есть осмысленный ответ

### Связь FR/NFR

| FR (что делает система) | NFR (как проверяем качество) |
|---|---|
| Создание мероприятия | Unit-тест на валидацию параметров |
| Регистрация участника | Тест на граничные условия (нет мест, невалидный email) |
| Отмена регистрации | Проверка, что место освобождается корректно |
| Все сценарии | Checkstyle + unit-тесты на критичную логику |

## Что строим / как проект эволюционирует

**Старт:** пустой репозиторий, JDK 21+, IDE.

**Результат:** консольное приложение — сервис записи на мероприятие.

Проект выбран не случайно: он достаточно прост для старта, но уже даёт доменную модель, правила валидации, управление ошибками и хранение состояния в коллекциях. Модуль 2 добавит Stream API и углубление в ООП, модуль 3 — Docker и БД, затем REST API, Kafka и observability.

**К концу модуля система умеет:**

| Capability | Что система делает |
|---|---|
| Управление мероприятиями | Создание с параметрами (название, лимит, дата), получение актуального состояния |
| Обработка заявок | Проверка правил (места, валидность данных), сохранение результата |
| Отмена регистрации | Освобождение места, пересчёт доступных слотов |
| Консольный интерфейс | Нумерованное меню, сбор ввода, отображение результатов и ошибок |
| Инженерное качество | Воспроизводимая сборка, тесты, style checks |

## Архитектура модуля 1: Service Layer + View Layer

Проект разделён на два явных слоя: **Service Layer** (бизнес-логика) и **View Layer** (консольный интерфейс). Это разделение готовит к Spring MVC / REST, где контроллеры не содержат бизнес-логику.

> Слой данных (`Repository`/`DAO`) добавляется в модуле 2 и модуле 3. В модуле 1 состояние хранится в памяти, в коллекциях.

### Service Layer

| Capability | Что обеспечивает |
|---|---|
| Управление мероприятиями | Создание мероприятия с параметрами, получение актуального состояния, проверка статуса регистрации |
| Обработка заявок | Приём заявки, проверка правил (места, валидные данные), сохранение результата |
| Отмена регистрации | Освобождение места, пересчёт доступных слотов |
| Валидация | Проверка входных данных и бизнес-правил (возраст, email, дата мероприятия) |

**Требования к слою:**

- Бизнес-правила инкапсулированы внутри слоя, не «утекают» в View
- Ошибки валидации возвращаются явно, не проглатываются
- Хранение in-memory (коллекции)
- Каждый сценарий покрыт unit-тестами

### View Layer — консольное меню

| Capability | Что обеспечивает |
|---|---|
| Отображение меню | Нумерованный список доступных действий |
| Ввод выбора | Чтение номера пункта и данных |
| Валидация ввода | Проверка, что номер соответствует существующему пункту |
| Отображение результатов | Вывод успешных результатов и ошибок в человекочитаемом формате |
| Делегирование | Передача выполнения в Service Layer, без бизнес-логики в View |

**Артефакт — структура меню:**

```text
=== Меню ===
1. Зарегистрировать участника
2. Показать участников
3. Добавить мероприятие
4. Показать детали мероприятия
5. Выход
```

**Ключевое правило:** View Layer **не содержит бизнес-логики**. Все проверки — внутри Service Layer. Меню только собирает ввод и делегирует выполнение сервисам.

## Как проверить результат

Модуль пройден, если:

- `./gradlew build` проходит без ошибок
- `java -jar build/libs/party-*.jar` запускается
- Реализованы возможности: создание мероприятия, регистрация, отмена, список участников
- Консольное меню отображается и позволяет выбрать действия
- Есть осмысленные тесты на правила регистрации и граничные кейсы
- Checkstyle настроен и не падает
