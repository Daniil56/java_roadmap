### Для кого этот модуль

**Ты прошёл модуль 1** и продолжаешь развивать проект — не создаёшь новый, а усиливаешь существующий: структуры данных, Stream API, CI/CD, AI-инструменты.

**Или ты не проходил модуль 1**, потому что посмотрел его содержание и понял, что базовый синтаксис, Gradle, тесты и Checkstyle тебе уже знакомы. В таком случае стартуй сразу отсюда — в начале модуля будет всё необходимое, чтобы встроиться в проект.

## Зачем этот модуль

Модуль 1 дал рабочую основу: консольный сервис, сборку, базовые тесты и стиль кода. Этого достаточно, чтобы стартовать, но проект пока "плоский" — все данные лежат в списках, бизнес-логика смешана с доступом к данным, а проверка качества работает только локально.

Цель модуля 2 — сделать проект инженерно взрослее. Ты не создаёшь новый проект, а усиливаешь существующий:

- Доменные сущности становятся выразительнее
- Доступ к данным по ключу перестаёт зависеть от линейных проходов
- Обработка коллекций становится осмысленной
- Проверка качества выходит в репозиторий через GitHub Actions

Каждое изменение модели данных — это ответ на бизнес-вопрос: зачем нужен лист ожидания, почему отмена последнего действия реализована через стек, какие trade-offs возникают при выборе структуры данных. При рефакторинге важно объяснять, **почему** модель изменилась, а не только **как** — trade-offs привязаны к контексту проекта.

AI на этом этапе подключается как **pair programmer** — концепция из Extreme Programming (XP): человек ведёт, AI помогает (тесты, рефакторинг, boilerplate). Подробно — в разделе AI SDLC ниже.

Общая рамка по FR/NFR, роли ИИ и логике маршрута — в [корневом README](../README.MD).

> Общая рамка: [README → Как работать с требованиями](../README.MD#как-работать-с-требования).

### Роли и контекст системы

**Чья это система:** Консольное приложение для **организатора** мероприятия.

Организатор управляет регистрацией участников: вводит данные, отменяет записи, ведёт очередь ожидания. Участники **не взаимодействуют** с системой напрямую — их данные вводит организатор.

Это важно для понимания требований:

- «регистрировать участника» → организатор вводит данные участника вручную
- «отменять регистрацию» → организатор отменяет регистрацию по запросу участника
- «отмена последнего действия» → организатор отменяет **свое** последнее действие (случайная ошибка ввода)
- «встать в лист ожидания» → организатор помещает участника в лист ожидания, когда мест нет

### Наследование из модуля 1

Все функциональные требования из модуля 1 **сохраняются и продолжают работать**:

- Создавать мероприятие с ограничением по участникам
- Регистрировать участника, если есть свободные места
- Отменять регистрацию и освобождать место
- Выводить список участников мероприятия

Модуль 2 не переписывает проект с нуля, а **усиливает** существующий код. Разница — в том, как реализовано хранение и обработка данных.

### Новые функциональные требования (модуль 2)

**Хранение и поиск:**

- Хранить и находить сущности по идентификатору за константное время O(1) — линейный поиск из модуля 1 больше не соответствует требованиям
- Доступ к данным отделён от бизнес-логики через Repository Layer

**Лист ожидания:**

Когда все места на мероприятие заняты, система не отказывает, а предлагает организатору поместить участника в очередь:

- Организатор помещает участника в конец очереди ожидания
- При отмене регистрации организатором первый из очереди автоматически занимает освободившееся место
- Порядок очереди строго FIFO: первый добавлен — первый получил место
- Если участник из очереди получил место, он удаляется из листа ожидания и появляется в основном списке участников

**Отмена последнего действия:**

- Система запоминает последнее действие организатора (регистрация, отмена, создание)
- Организатор может отменить своё последнее действие одной командой — например, если случайно зарегистрировал не того участника
- Отмена работает по принципу LIFO: отменяется то, что было сделано последним
- Система показывает, какое именно действие будет отменено, перед подтверждением

**Обработка данных:**

- Фильтровать мероприятия по дате, месту, возрастному ограничению
- Сортировать участников по имени, дате регистрации
- Группировать мероприятия по месту проведения

**Ошибки:**

- Различать типы ошибок через иерархию исключений (например, `EntityNotFoundException`, `DuplicateException`, `CapacityExceededException`)
- Каждое исключение содержит контекст: какой ID, какая операция

**CI/CD:**

- Проходить автоматическую проверку сборки и тестов в GitHub Actions при каждом push/PR
- Показывать покрытие тестами через JaCoCo

### Алгоритмические требования

| Операция | Требуемая сложность | Какая структура помогает |
|---|---|---|
| Поиск по ID | O(1) | `HashMap` |
| Добавление | O(1) | `HashMap`, `ArrayList` (амортизированно) |
| Удаление | O(1) | `HashMap` |
| Фильтрация, сортировка, группировка | O(n) | `Stream API` по коллекции |
| Добавление в очередь ожидания | O(1) | `Queue` (LinkedList как реализация) |
| Извлечение первого из очереди | O(1) | `Queue.poll()` |
| Запомнить действие для отмены | O(1) | `Stack.push()` |
| Отменить последнее действие | O(1) | `Stack.pop()` |

### Опциональные требования (со звёздочкой)

Эти задания углубляют понимание внутренних механизмов стандартных коллекций. Реализовав собственную структуру, ты понимаешь, почему стандартная работает именно так.

- Реализовать собственные структуры данных (`SimpleArrayList`, `SimpleLinkedList`, `SimpleHashMap`, `SimpleHashSet`) вместо стандартных коллекций
- Использовать реализованные структуры в проекте: `SimpleHashMap` для Repository, `SimpleLinkedList` для листа ожидания, собственный Stack для отмены действий
- Читать настройки из внешнего файла конфигурации (формат `Properties`)
- Сохранять данные персистентно в CSV-файлах: перезапуск приложения не должен приводить к потере данных. Для хранения и чтения файлов использовать собственную реализацию списка вместо стандартного `ArrayList`

### Практическая связь FR/NFR

| Что делаем | Как проверяем |
|---|---|
| Хранение по ключу O(1) | Тест: поиск по ID не зависит от размера данных |
| Лист ожидания | Тест: при отмене первый из листа ожидания получает место автоматически |
| Отмена последнего действия | Тест: после регистрации отмена отменяет именно её |
| Обработка через Stream | Тесты на фильтрацию, сортировку, группировку |
| Иерархия исключений | Негативные тесты на каждый тип ошибки |
| Автоматическая проверка | Зелёный GitHub Actions на push/PR |
| Покрытие тестами | JaCoCo отчёт, слабые места закрыты |

> Каждый FR привязан к бизнес-контексту проекта: лист ожидания — это не «очередь на структурах данных», а ответ на вопрос «что происходит, когда места закончились?». Отмена последнего действия — не «стек LIFO», а «организатор ошибся и хочет вернуть всё как было». Trade-offs при выборе структур данных объясняются через контекст проекта.

## Нефункциональные требования

**Как проверяем качество:**

- `./gradlew clean build` проходит локально
- `./gradlew test` проходит локально и в GitHub Actions
- `./gradlew jacocoTestReport` генерирует отчёт
- Код читаем: структуры данных выбраны по задаче
- Ошибки выражены через иерархию исключений

**CI/CD pipeline:**

- Pipeline должен содержать отдельные job'ы (стадии): checkstyle, build, test
- Каждый job должен выполняться независимо и показывать свой статус
- При падении на одной стадии должно быть понятно, какая именно проверка не прошла

**AI SDLC (AI Pair Programming):**

- Начиная с этого модуля AI выступает как **pair programmer** — концепция из Extreme Programming: два разработчика работают вместе, один ведёт (человек), другой помогает (AI). AI может генерировать тесты и рефакторить код, но человек проверяет результат через метрики покрытия (JaCoCo), читаемость тестов и полноту проверяемых сценариев
- CLI-агент может писать тесты и выполнять рефакторинг — рекомендация, а не требование. Цель — научиться работать с AI как с партнёром, а не только как с проводником по концепциям
- Если AI-сгенерированный код непонятен — это сигнал остановиться и попросить агента описать, что делает код и почему. Понимание важнее скорости
- Inline-подсказки (Copilot) принимаются только если полностью понятны
- AI используется для code review, навигации по документации и генерации рутинного кода (тесты, boilerplate) — бизнес-логика по-прежнему пишется руками

## Архитектура модуля 2: Repository Layer

В модуле 1 было два слоя: **Service Layer** (бизнес-логика) и **View Layer** (консольное меню). В модуле 2 появляется третий слой — **Repository Layer**.
**Что должен уметь слой:**

| Capability | Что система делает |
|------------|-------------------|
| Хранение по ключу | Находит сущность по ID за O(1) |
| Управление уникальностью | Гарантирует, что ключ не дублируется |
| CRUD-операции | Создаёт, читает, обновляет, удаляет сущности |

**Требования к реализации:**

- Доступ к данным отделён от бизнес-логики
- Сервисы не знают, как именно хранятся данные — только через интерфейс репозитория
- Хранение in-memory (коллекции), без БД — БД появится в модуле 4

**Ключевое правило:** Service Layer не работает с коллекциями напрямую — только через Repository.
